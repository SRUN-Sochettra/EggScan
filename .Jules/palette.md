## 2024-06-10 - Adding form input and modal close button accessibility
**Learning:** Found an accessibility issue pattern specific to this app's components: custom inputs with absolute positioned icons/text (like the `@` symbol in `ScanForm` and `BattleForm`) often lack explicit `aria-label`s because they rely on visual placeholders or adjacent layout elements rather than semantic labels. Also, the `Leaderboard` modal used a standard text "✕" as an icon-only button without an `aria-label`.
**Action:** Always verify that inputs with decorative/absolute-positioned elements still have proper `aria-label`s or associated `<label>`s, and ensure all icon-only or decorative-text-only buttons (like "✕") have clear `aria-label`s describing their action.
## 2024-06-11 - Converting interactive div list items to semantic a tags for keyboard accessibility
**Learning:** Found a UX/accessibility issue where list items in a generic container (`div`) were acting as links by overriding the `onClick` handler with a programmatic routing combined with `window.location.reload()`. This prevents keyboard navigation (tabbing), standard context menus ("Open in new tab"), and removes screen reader semantics. Additionally, interactive elements lacked `focus-visible` styles for tabbing visibility.
**Action:** Always replace `div` based pseudo-links with native `<a>` tags with `href` attributes, and ensure that focus outlines are provided for keyboard navigation using utilities like `focus-visible:ring-2 focus-visible:ring-brown-500`.
